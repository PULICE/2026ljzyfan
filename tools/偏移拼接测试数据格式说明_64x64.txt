========================================================================
UVSP Full 协议 - 偏移拼接测试数据格式说明
========================================================================
分辨率: 64x64  像素格式: RAW10_PACKED (bayer_bggr8, 1 byte/pixel)
帧大小: 64 × 64 × 1 = 4096 字节
包数量: 4 个 UDP 包
单个 UDP 包最大载荷: 1280 字节（协议规定）

========================================================================
一帧的 4 个包概览
========================================================================

+--------+-------+-------+-------+-----------+--------------------------+
| 包序号 | seq   | flags | TLV数 | payload   | CHUNK_OFFSET             |
+--------+-------+-------+-------+-----------+--------------------------+
| #1     | 1     | 0x02  | 13    | ~1160B    | 0                        |
| #2     | 2     | 0x00  | 3     | ~1240B    | 1160                     |
| #3     | 3     | 0x00  | 3     | ~1240B    | 2400                     |
| #4     | 4     | 0x00  | 3     | ~1240B    | 3640                     |
+--------+-------+-------+-------+-----------+--------------------------+

每个包都携带 CHUNK_OFFSET TLV，解析器按偏移直接写入帧缓冲区，
不依赖包的到达顺序。


========================================================================
包 #1 (seq=1) — SOF 包 (flags=0x02 SOF 位=1)
========================================================================
SOF 位=1 表示新帧开始，同时携带完整的帧描述信息。

┌─────────────────────────────────────────────────────────────────────────┐
│ BaseHeader (8 bytes)                                                    │
├──────────┬──────────────┬───────────┬──────────┬────────────────────────┤
│ 偏移     │ 字段          │ 值        │ 大小     │ 说明                   │
├──────────┼──────────────┼───────────┼──────────┼────────────────────────┤
│ 0x00~0x01│ magic        │ 0x5654    │ 2B       │ UVSP Full 协议标识      │
│ 0x02     │ flags        │ 0x02      │ 1B       │ SOF=0x02               │
│          │              │ (或 0x22) │          │ SOF|DESC=0x22 变更时   │
│ 0x03     │ profile      │ 0x00      │ 1B       │ 保留/配置               │
│ 0x04~0x05│ source_id    │ 0x0001    │ 2B       │ 源 ID = 1              │
│ 0x06~0x07│ type         │ 0x0000    │ 2B       │ 类型                    │
├──────────┴──────────────┴───────────┴──────────┴────────────────────────┤
│ CommonTransport (8 bytes)                                              │
├──────────┬──────────────┬───────────┬──────────┬────────────────────────┤
│ 0x08~0x09│ seq_num      │ 0x0001    │ 2B       │ 序列号 = 1             │
│ 0x0A~0x0D│ timestamp    │ 见数据     │ 4B       │ 时间戳 (大端)           │
│ 0x0E     │ ext_count    │ 0x0D      │ 1B       │ TLV 数量 = 13           │
│ 0x0F     │ reserved     │ 0x00      │ 1B       │ 保留                    │
├──────────┴──────────────┴───────────┴──────────┴────────────────────────┤

绝对时间戳 (8 bytes, 可选)
  - 位置: CommonTransport 之后、TLV 扩展段之前。
  - 编码: uint64, 大端。
  - 触发: flags.TS=0x10 且包内实际存在该 8B 字段时解析。
  - 兼容: 当前真实设备可能设置 TS 位但未插入 8B 绝对时间戳；解析器会同时校验
    “无绝对时间戳”和“有绝对时间戳”两种 TLV 起始偏移，避免误跳过 TLV。

├──────────┴──────────────┴───────────┴──────────┴────────────────────────┤
│ TLV 扩展段 (13 个 TLV, 4 字节对齐，共 104 字节)                         │
├──────────┬──────────┬──────────┬────────────────────────────────────────┤
│ TLV#     │ type     │ len      │ value (u32, 大端)                      │
├──────────┼──────────┼──────────┼────────────────────────────────────────┤
│ TLV#1    │ 0x01     │ 4        │ FRAME_ID      = 0x00000001             │
│ TLV#2    │ 0x02     │ 4        │ CHUNK_OFFSET  = 0x00000000 ← 首包偏移=0 │
│ TLV#3    │ 0x03     │ 4        │ CHUNK_LENGTH  = 本块实际长度 (≈1160)    │
│ TLV#4    │ 0x04     │ 4        │ TOTAL_SIZE    = 0x00001000 (4096)      │
│ TLV#5    │ 0x05     │ 4        │ PIXFMT        = 5  ← RAW10_PACKED     │
│ TLV#6    │ 0x06     │ 4        │ WIDTH         = 64                     │
│ TLV#7    │ 0x07     │ 4        │ HEIGHT        = 64                     │
│ TLV#8    │ 0x08     │ 4        │ 填充数据 (0x01)                        │
│ TLV#9    │ 0x0b     │ 4        │ 填充数据 (0x01)                        │
│ TLV#10   │ 0x0c     │ 4        │ 填充数据 (33333)                       │
│ TLV#11   │ 0x20     │ 4        │ 填充数据 (0x01)                        │
│ TLV#12   │ 0x21     │ 4        │ 填充数据 (640)                         │
│ TLV#13   │ 0x22     │ 4        │ 填充数据 (480)                         │
├──────────┴──────────┴──────────┴───────────────────────────────────────┤
│ Payload (≈1160 字节) — 图像数据偏移 0 ~ 1159                             │
│ 棋盘格: 8x8 黑白块交替                                                   │
└────────────────────────────────────────────────────────────────────────┘


========================================================================
包 #2~#4 (seq=2~4) — 非 SOF 包 (flags=0x00, ext_count=3)
========================================================================

┌─────────────────────────────────────────────────────────────────────────┐
│ BaseHeader (8 bytes)     magic=0x5654, flags=0x00, source_id=1         │
├─────────────────────────────────────────────────────────────────────────┤
│ CommonTransport (8 bytes)   seq=2~4, ext_count=3                       │
├─────────────────────────────────────────────────────────────────────────┤
│ TLV 扩展段 (3 个 TLV, 4 字节对齐，共 24 字节)                           │
├──────────┬──────────┬──────────┬────────────────────────────────────────┤
│ TLV#1    │ 0x02     │ 4        │ CHUNK_OFFSET = 包对应的帧内偏移        │
│ TLV#2    │ 0x08     │ 4        │ 填充数据 (0x01)                       │
│ TLV#3    │ 0x0b     │ 4        │ 填充数据 (0x01)                       │
├──────────┴──────────┴──────────┴────────────────────────────────────────┤
│ Payload (≈1240 字节)                                                  │
└─────────────────────────────────────────────────────────────────────────┘

包 #2: seq=2, CHUNK_OFFSET=1160,  Payload=图像数据偏移 1160 ~ 2399
包 #3: seq=3, CHUNK_OFFSET=2400,  Payload=图像数据偏移 2400 ~ 3639
包 #4: seq=4, CHUNK_OFFSET=3640,  Payload=图像数据偏移 3640 ~ 4095


========================================================================
PIXFMT 枚举值（协议定义 0x05 TLV）
========================================================================

| 值 | 名称          | 每像素字节 | ffplay pixel_format |
|----|---------------|----------|---------------------|
| 1  | YUYV422       | 2        | yuyv422             |
| 2  | RGB565        | 2        | rgb565              |
| 3  | NV12          | 1.5      | nv12                |
| 4  | GREY8         | 1        | gray                |
| 5  | RAW10_PACKED  | 1        | bayer_bggr8         |
| 6  | RAW12_PACKED  | 1        | bayer_bggr8         |

注意: 当前测试脚本 PIXFMT = 5 (RAW10_PACKED)，对应 bayer_bggr8。


========================================================================
flags 位定义
========================================================================

| Bit | 名称  | 值    | 说明                                              |
|-----|-------|-------|---------------------------------------------------|
| 7-6 | -     | -     | 保留                                              |
| 5   | DESC  | 0x20  | 描述/尺寸变更。TOTAL_SIZE/PIXFMT/尺寸等发生变化。   |
|     |       |       | 仅在 SOF=1 的首个分片中有意义。                      |
| 4   | TS    | 0x10  | Timestamp 有效标志                                 |
| 3   | RST   | 0x08  | 复位                                             |
| 2   | GAP   | 0x04  | 帧间隔标志，Payload 可忽略                         |
| 1   | SOF   | 0x02  | 单元起始（新帧开始）                                |
| 0   | EOF   | 0x01  | 单元结束                                          |

DESC 位使用场景:
  - 分辨率从 64x64 变为 128x128 时, SOF 包 flags = 0x22 (SOF|DESC)
  - 解析器收到 DESC 后标记 pending_reconfig，下一帧 SOF 重新分配缓冲区
  - 即使上一帧还没收完，也不影响下一帧的分辨率变更


========================================================================
动态分辨率变更处理（DESC 位工作机制）
========================================================================

初始帧 (64x64, 4 个包/帧):
  包 1 (SOF)           → 拼帧中 (filled=1160/4096)
  包 2 (offset=1160)   → 拼帧中 (filled=2400/4096)
  包 3 (offset=2400)   → 拼帧中 (filled=3640/4096)

下一帧 (128x128, SOF|DESC):
  包 1 (SOF|DESC, flags=0x22)
    TOTAL_SIZE=16384, WIDTH=128, HEIGHT=128, PIXFMT=5
    → pending_reconfig = true (标记重配置)
  包 2 (offset=1160)   → 但缓冲区大小还是原来的 4096? 
    → 回退到旧帧拼接，但这帧可能不全

再下一帧 (128x128, SOF, 无 DESC):
  包 1 (SOF, flags=0x02)
    pending_reconfig = true → 用新 TOTAL_SIZE=16384 重新分配 buf
    → pending_reconfig = false
    → 开始正常拼接 128x128 帧

注意: 解析器在收到 DESC 位的 SOF 包时不会立即重新分配，
      因为当前帧可能还在接收过程中。等下一个不带 DESC 的
      SOF 包到来时再做重配置，确保不干扰当前帧的接收。


========================================================================
关键设计要点
========================================================================

1. 每个包都有 TLV
   ─────────────────
   即使是中间包 (非 SOF) 也携带 ext_count=3 的 TLV 扩展，
   其中第一个 TLV 就是 CHUNK_OFFSET=实际偏移。
   解析器根据每个包的偏移信息，把数据直接写到帧缓冲区的正确位置，
   不依赖按顺序到达。

2. SOF 包携带完整元信息
   ─────────────────────
   SOF 包携带了 WIDTH, HEIGHT, TOTAL_SIZE, PIXFMT 等完整信息，
   解析器在收到 SOF 时即可知道帧的大小和格式，无需配置。

3. 数据排列模式
   ──────────────
   棋盘格 (checkerboard):
   每个 8x8 块交替黑(32)和白(255)

   第 0~7 行: [255,255,...,32,32,255,255,...,32,32,...]  以 8 像素为周期
   第 8~15 行: [32,32,...,255,255,32,32,...,255,255,...]  颜色翻转


========================================================================
与真实设备数据的差异
========================================================================

| 项目            | 本测试脚本              | 你的真实设备            |
|-----------------|-------------------------|------------------------|
| TLV 数量 (SOF)  | 13 个                   | 约 16 个（原日志显示）  |
| TLV 数量 (非SOF)| 3 个                    | 3 个                   |
| WIDTH/HEIGHT    | ✅ 有 (TLV 0x06/0x07)  | ❌ 没有 (w=0 h=0)      |
| TOTAL_SIZE      | ✅ 有 (TLV 0x04)        | ❌ 没有 (total_size=0) |
| CHUNK_OFFSET    | ✅ 每个包都有            | ✅ 每个包都有           |
| PIXFMT          | ✅ 5 (RAW10_PACKED)     | ❓ 待确认              |
| 像素格式        | bayer_bggr8 (1B/px)    | 根据设备配置            |
| 图像内容        | 棋盘格                  | 真实摄像头图像           |
| 包间隔          | 1ms (模拟网络延迟)      | 实际网络延迟             |

真实设备缺少 WIDTH/HEIGHT/TOTAL_SIZE TLV，
解析器通过 g_cfg 配置（main.cpp）和 FRAME_SIZE 宏来兜底。
